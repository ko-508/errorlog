## エラーの概要

PII-Shieldは、ログから個人情報（PII: Personally Identifiable Information）や機密情報（シークレット）を削除するためのオープンソースツールです。このツールは、ログがアプリケーションから出力される「前」にデータをサニタイズすることで、PIIがログ収集システム（ELKスタック、Loki、SIEMなど）に到達するリスクを最小限に抑えます。典型的な発生状況としては、アプリケーションログにメールアドレス、電話番号、パスワード、APIキー、クレジットカード番号などの機密情報が含まれてしまい、これらが意図せず外部システムに流出するのを防ぐ目的で使用されます。

## 実際のエラーメッセージ例

PII-Shieldは、ログのサニタイズに成功した場合、エラーメッセージを出力するのではなく、機密情報を `[HIDDEN:ハッシュ値]` の形式に置き換えます。以下は、PII-Shieldが適用されたログの出力例です。

```
login failed email=[HIDDEN:a1b2c3] password=[HIDDEN:d4e5f6]
```

```json
{
  "timestamp": "2023-10-27T10:00:00Z",
  "level": "ERROR",
  "message": "Payment processing error",
  "user_id": "user-123",
  "transaction_id": "txn-789",
  "card_number": "[HIDDEN:g7h8i9]",
  "api_key": "[HIDDEN:j0k1l2]"
}
```

上記のように、元のログに含まれていたメールアドレス、パスワード、クレジットカード番号、APIキーなどが、特定のハッシュ値を含む `[HIDDEN:...]` 形式に置換されています。これにより、機密情報がログに直接記録されることを防ぎつつ、デバッグに必要な情報の関連付け（同じ値が同じハッシュになる）を維持できます。

## よくある原因と解決手順

PII-Shieldが期待通りに機能しない、または設定が不十分な場合、機密情報が漏洩する可能性があります。ここでは、よくある原因とその解決手順を説明します。

### 原因1：機密キーの検出漏れ

PII-Shieldは、`password=...` や `token=...` のような特定のキーを検出してその値を隠蔽します。しかし、アプリケーションが独自のキー名（例: `auth_secret=...`）を使用している場合、デフォルト設定では検出されません。

**Before（エラーが起きるコード）：**

```yaml
# PII-Shieldのデフォルト設定（カスタムキーが考慮されていない）
# PII_CUSTOM_REGEX_LIST が未設定
```

**After（修正後）：**

```bash
# 環境変数でカスタムキーを定義する例
export PII_CUSTOM_REGEX_LIST='[
  {"pattern": "auth_secret=([a-zA-Z0-9_\\-]+)", "name": "AuthSecret"},
  {"pattern": "internal_id=([0-9]{10})", "name": "InternalID"}
]'
```

**解決手順:**
アプリケーションが使用する機密情報を含むキー名を特定し、`PII_CUSTOM_REGEX_LIST` 環境変数または設定ファイルでカスタム正規表現ルールとして追加します。これにより、PII-Shieldはこれらのキーに続く値を機密情報として認識し、適切に隠蔽します。

### 原因2：特定のデータ型（クレジットカード番号など）の検出漏れ

PII-ShieldはLuhnアルゴリズムなどを用いてクレジットカード番号のような特定のデータ型を検出しますが、文脈によっては誤検出を避けるために検出されない場合があります。また、企業固有のID形式（例: 顧客ID、医療記録番号）は、デフォルトでは検出されません。

**Before（エラーが起きるコード）：**

```yaml
# PII-Shieldのデフォルト設定（特定の企業固有IDが考慮されていない）
# PII_CUSTOM_REGEX_LIST が未設定
```

**After（修正後）：**

```bash
# 環境変数でカスタム正規表現ルールを定義する例
export PII_CUSTOM_REGEX_LIST='[
  {"pattern": "MRN-[0-9]{8}", "name": "MedicalRecordNumber"},
  {"pattern": "CUST-[A-Z]{3}-[0-9]{5}", "name": "CustomerIdentifier"}
]'
```

**解決手順:**
アプリケーションがログに出力する企業固有のIDや、特定の形式を持つ機密情報（例: `MRN-12345678`）がある場合、それらを検出するためのカスタム正規表現ルールを `PII_CUSTOM_REGEX_LIST` に追加します。これにより、これらの情報も適切に隠蔽されます。

### 原因3：エントロピーベースの検出の誤判定

PII-Shieldは、エントロピー（ランダム性）が高い文字列をシークレットとして検出する機能を持っています。しかし、コミットハッシュ、UUID、トレースID、リクエストIDなど、ランダムに見えるが機密情報ではない値も存在します。これらが誤って隠蔽されると、デバッグが困難になります。

**Before（エラーが起きるコード）：**

```yaml
# PII_SAFE_REGEX_LIST が未設定で、UUIDやトレースIDがHIDDENになる
```

**After（修正後）：**

```bash
# 環境変数で安全な正規表現リストを定義する例
export PII_SAFE_REGEX_LIST='[
  {"pattern": "^[a-f0-9]{7}$", "name": "GitShortSHA"},
  {"pattern": "^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$", "name": "UUID"},
  {"pattern": "^trace-[0-9a-f]{16}$", "name": "TraceID"}
]'
```

**解決手順:**
誤って機密情報として扱われる可能性のある、ランダムに見えるが安全な文字列（例: UUID、Gitコミットハッシュ、トレースID）を `PII_SAFE_REGEX_LIST` に追加します。このリストのルールは他のチェックよりも優先されるため、これらの値が誤って隠蔽されるのを防ぎ、ログの有用性を維持できます。

## ツール固有の注意点

PII-Shieldは、ログのサニタイズをアプリケーションの「近く」で行うことを重視しています。これは、FluentdやLogstashなどのログパイプラインで処理するよりも早い段階でPIIを削除することで、中間システムでの漏洩リスクを低減するためです。

*   **Kubernetesでのサイドカーパターン:** PII-Shieldの主な利用方法は、Kubernetesのサイドカーコンテナとしてデプロイすることです。アプリケーションコンテナが共有ボリュームにログを書き込み、PII-Shieldサイドカーがそのファイルを読み取り、サニタイズ後に標準出力に書き出します。この設計により、アプリケーションコードを変更することなくPII保護を導入できます。
*   **パフォーマンスへの配慮:** PII-Shieldは、JSONログを処理する際に、完全なJSONパーサーではなく、必要な部分だけを解析する軽量なパーサーを使用しています。これにより、ログストリーム処理におけるオーバーヘッドを最小限に抑え、高負荷時でも安定したパフォーマンスを維持します。
*   **`[HIDDEN:ハッシュ値]` の利用:** 単に `[REDACTED]` と表示するのではなく、元の値のハッシュ値を含めることで、同じ機密情報がログ内で同じハッシュ値を持つようになります。これにより、元の値を知ることなく、デバッグ時にイベントの関連付けや相関分析を行うことが可能になります。`PII_SALT` 環境変数でソルトを固定することで、この一貫性を維持できます。
*   **Fail Open / Fail Closed:** PII-Shieldは、スキャナーが失敗した場合の挙動として「Fail Open」（ログをそのまま通過させる）と「Fail Closed」（ログを破棄または代替マーカーに置き換える）を選択できます。プライバシー保護を最優先する場合はFail Closed、システムの可用性や監視を優先する場合はFail Openを選択するなど、ユースケースに応じて適切なポリシーを設定することが重要です。

## それでも解決しない場合

PII-Shieldの設定や挙動に問題があり、上記の手順で解決しない場合は、以下の点を確認してください。

*   **PII-Shieldのログ:** PII-ShieldサイドカーコンテナやCLIツールのログを確認し、エラーメッセージや警告が出力されていないか確認します。特に、設定ファイルのパースエラーや正規表現のコンパイルエラーなどがないか注意深く確認してください。
    *   Kubernetesの場合: `kubectl logs <pii-shield-sidecar-pod-name> -c pii-shield`
*   **設定の適用状況:** 環境変数や設定ファイルが正しくPII-Shieldに渡されているかを確認します。特にKubernetes環境では、`PiiPolicy` リソースやDeploymentのアノテーションが正しく適用されているか、`kubectl describe pod <your-app-pod>` でサイドカーコンテナの環境変数を確認します。
*   **正規表現のテスト:** カスタム正規表現が意図した文字列にマッチするかどうかを、PII-Shieldの外部で（例: regex101.comなどのオンラインツールや `grep -P` コマンドで）テストします。正規表現が複雑すぎたり、誤っていたりすると、期待通りに動作しないことがあります。
*   **公式ドキュメントの参照:** PII-Shieldの公式GitHubリポジトリ（https://github.com/pii-shield/pii-shield）やドキュメントを参照し、最新の情報や詳細な設定オプションを確認します。特に、新しいバージョンで追加された機能や変更点がないか確認してください。
*   **デバッグモードの活用:** PII-Shieldにデバッグオプションがある場合、それを有効にして詳細な処理ログを出力させ、どのルールが適用され、どの値がどのように処理されたかを確認します。
