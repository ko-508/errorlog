## エラーの概要

この記事では、JVM依存関係のサプライチェーンにおける異常を検知する振る舞いスキャナー「Marshal」の知見について解説します。Marshalは、CVE（共通脆弱性識別子）に依存せず、パッケージの変更履歴（メンテナーの変更、署名の欠落、リポジトリURLの変更など）を監視し、リスクスコアを付与することで、潜在的なサプライチェーン攻撃をPR（プルリクエスト）段階でブロックすることを目指しています。

## 実際のエラーメッセージ例

Marshalは、直接的なHTTPエラーメッセージを出力するツールではありませんが、リスクを検知した際には以下のような形式で警告やスコアを提示します。

**Marshal CLI出力例（想定）:**

```
Scanning dependency: javax.activation:activation
  Version: 1.1.1
  Risk Score: 55/100 (ORANGE)
  Signals:
    - SIGNATURE_DROPPED (40 pts): Version 1.1-rev-1 was signed with GPG key2D6641C6AF88103E (Sun Microsystems). Version 1.1.1 has no .asc file.
    - MISSING_SIGNATURE (15 pts): Current release is unsigned.
```

**GitHub Actionsコメント例（想定）:**

```
🚨 Marshal Security Scan Alert 🚨

Dependency: log4j:log4j (version 1.2.17)
Risk Score: 55/100 (ORANGE)

Potential anomalies detected:
- NEW_MAINTAINER: Signing key changed from D3EC499070C9C3D0 to 86E02C5A42196CA8.
- REPO_CHANGED: SCM URL changed from .../tags/v1_2_16 to .../tags/v1_2_17_rc3.

Please review these changes carefully before merging.
```

## よくある原因と解決手順

Marshalが検知する「異常」は、必ずしも悪意のある攻撃とは限りません。多くの場合、正当な理由による変更がリスクとしてフラグ付けされることがあります。ここでは、Marshalが検知する主な原因と、それに対する考え方や解決手順を説明します。

### 原因1：署名の欠落または変更

パッケージの署名が以前のバージョンから欠落している、または署名キーが変更されている場合に発生します。これは、アカウント乗っ取り後の攻撃者が元の署名キーを持たない場合に起こりうる典型的なシナリオです。

**なぜ発生するか：**
正当な理由としては、企業の買収（例：Sun MicrosystemsからOracleへのJava EEコンポーネントの移行）、プロジェクトのメンテナー変更、または署名インフラストラクチャの更新などが考えられます。攻撃の場合、攻撃者は元の署名キーを持たないため、新しいリリースは署名なしで公開されるか、異なるキーで署名されます。

**Before（エラーが起きるコード）：**

```xml
<!-- pom.xml で署名が欠落した、または異なるキーで署名された依存関係を使用 -->
<dependency>
    <groupId>javax.activation</groupId>
    <artifactId>activation</artifactId>
    <version>1.1.1</version> <!-- 以前のバージョンは署名されていたが、このバージョンは署名なし -->
</dependency>
```

**After（修正後）：**

このケースでは、コードの修正ではなく、状況の評価と対応が重要です。

```
# 修正後の対応例：
# 1. Marshalの警告をレビューし、公式アナウンスやプロジェクトの変更履歴を確認する。
# 2. 企業の買収やメンテナー変更など、正当な理由がある場合は、そのリスクを受け入れるか、代替ライブラリを検討する。
# 3. 不明な変更の場合は、その依存関係の使用を一時停止し、詳細な調査を行う。
# 4. Marshalのルールを調整し、既知の正当な変更（例：Oracleによるjavax.*アーティファクトの署名変更）をホワイトリスト化する。
```

### 原因2：メンテナー（署名キー）の変更

パッケージの署名キーが以前のバージョンから変更されている場合に発生します。これは、新しいメンテナーがプロジェクトを引き継いだり、組織が署名インフラストラクチャを更新したりする際に起こります。

**なぜ発生するか：**
正当な理由としては、Apache Software Foundationのような大規模な組織が、複数のプロジェクトにわたって署名インフラストラクチャを定期的にローテーションすることが挙げられます。攻撃の場合、悪意のあるアクターがプロジェクトの制御を奪い、自身のキーで署名する可能性があります。

**Before（エラーが起きるコード）：**

```xml
<!-- pom.xml で署名キーが変更された依存関係を使用 -->
<dependency>
    <groupId>log4j</groupId>
    <artifactId>log4j</artifactId>
    <version>1.2.17</version> <!-- 以前のバージョンから署名キーが変更されている -->
</dependency>
```

**After（修正後）：**

このケースも、コードの修正ではなく、状況の評価とMarshalのルール調整が重要です。

```
# 修正後の対応例：
# 1. Marshalの警告をレビューし、新しい署名キーが信頼できる組織（例：Apache Software Foundation）に属しているかを確認する。
#    - GPGキーのフィンガープリントを検索し、公式情報と照合する。
# 2. 組織内でキーローテーションが頻繁に行われる場合は、Marshalのルールを調整し、
#    同じ組織に属するキー間の変更はリスクとして扱わないように設定する。
#    （例：Marshal v0.2で導入予定の「同じ組織に属するキーかどうかのチェック」）
# 3. 不明なキーへの変更の場合は、その依存関係の使用を一時停止し、詳細な調査を行う。
```

### 原因3：リポジトリURLの変更

パッケージのSCM（ソースコード管理）リポジトリURLが以前のバージョンから変更されている場合に発生します。

**なぜ発生するか：**
正当な理由としては、プロジェクトが異なるリポジトリホスティングサービスに移行したり、組織内でリポジトリ構造が変更されたりする場合が考えられます。攻撃の場合、攻撃者が悪意のあるリポジトリに誘導するためにURLを変更する可能性があります。

**Before（エラーが起きるコード）：**

```xml
<!-- pom.xml でSCM URLが変更された依存関係を使用 -->
<dependency>
    <groupId>commons-collections</groupId>
    <artifactId>commons-collections</artifactId>
    <version>3.2.2</version> <!-- 以前のバージョンからSCM URLが変更されている -->
</dependency>
```

**After（修正後）：**

このケースも、コードの修正ではなく、状況の評価とMarshalのルール調整が重要です。

```
# 修正後の対応例：
# 1. Marshalの警告をレビューし、新しいリポジトリURLが元の組織と同じドメインまたはGitHub組織に属しているかを確認する。
#    - 例：同じApache SVNドメイン内でのパス変更は低リスク。
#    - 例：github.com/org-name/repo から github.com/random-user/repo への変更は高リスク。
# 2. Marshalのルールを調整し、同じドメイン内でのURL変更はリスクとして扱わないように設定する。
#    （例：Marshal v0.2で導入予定の「同じドメイン認識」）
# 3. 異なるドメインや信頼できないユーザーへの変更の場合は、その依存関係の使用を一時停止し、詳細な調査を行う。
```

## ツール固有の注意点

Marshalのような振る舞いベースのスキャナーは、従来のCVEデータベースでは検知できない潜在的な脅威を表面化させる強力なツールです。しかし、その特性上、多くの「誤検知（False Positive）」が発生する可能性があります。

*   **コンテキストの理解が重要:** Marshalが示すリスクスコアやシグナルは、あくまで「異常」を示唆するものであり、直ちに「攻撃」を意味するわけではありません。検知された異常が、企業の買収、組織内のインフラ変更、メンテナーの交代といった正当な理由によるものなのか、それとも実際の攻撃の兆候なのかを、人間がコンテキストを理解して判断する必要があります。
*   **ルールの継続的な改善:** Marshalは、検知ロジックを継続的に改善していく必要があります。例えば、署名キーの変更が同じ組織内で行われたものか、リポジトリURLの変更が同じドメイン内で行われたものかを識別する機能は、誤検知を減らす上で非常に重要です。ユーザーは、ツールのアップデートに注目し、最新のルールセットを利用することが推奨されます。
*   **既存のセキュリティ対策との併用:** Marshalは、既存の脆弱性スキャナー（例：OWASP Dependency-Check）や静的コード解析ツールと組み合わせて使用することで、より多層的なセキュリティ対策を実現できます。Marshalは「振る舞い」の変化に焦点を当て、CVEベースのツールでは見逃されがちな脅威を補完します。

## それでも解決しない場合

Marshalの警告が正当なものか判断に迷う場合や、誤検知が頻繁に発生して運用に支障が出る場合は、以下の手順で対処してください。

1.  **詳細なログの確認:** Marshalが提供する詳細なスキャンログやJSON出力を確認し、どのシグナルが、どのような具体的な変更に基づいてトリガーされたのかを深く理解します。
2.  **プロジェクトの公式情報を参照:** 警告対象の依存関係の公式ドキュメント、GitHubリポジトリ、リリースノート、メーリングリストなどを確認し、関連する変更（メンテナー変更、リポジトリ移行、署名ポリシー変更など）に関する公式アナウンスがないかを調べます。
3.  **GPGキーの検証:** 署名キーの変更が検知された場合、新しいGPGキーのフィンガープリントを信頼できるキーサーバーで検索し、そのキーがどの組織に属しているか、信頼できるものかを検証します。
4.  **Marshalの公式ドキュメントとコミュニティ:** Marshalの公式ドキュメント（公開され次第）を参照し、設定オプションやルールの調整方法について確認します。また、MarshalのコミュニティやIssueトラッカーで同様の問題が報告されていないかを確認し、必要であれば質問を投稿します。
5.  **ツールの設定調整:** Marshalが提供する設定オプション（例：特定のシグナルの感度調整、特定の依存関係のホワイトリスト化）を利用して、プロジェクトの特性に合わせたチューニングを試みます。
