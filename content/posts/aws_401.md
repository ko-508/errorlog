## エラーの概要

AWSで401[エラー](/glossary/エラー/)が返される場合、[リクエスト](/glossary/リクエスト/)に含まれる認証情報が無効であることを示しています。AWS [API](/glossary/api/)、[SDK](/glossary/sdk/)、[CLI](/glossary/cli/)のいずれかを使用する際に、アクセスキー、シークレットアクセスキー、セッショントークン、または[IAM](/glossary/iam/)[ロール](/glossary/ロール/)の認証情報が不正または期限切れの状態で送信されると発生します。この[エラー](/glossary/エラー/)は認証層での問題であり、比較的簡単に解決できるケースがほとんどです。

## 実際のエラーメッセージ例

AWS [CLI](/glossary/cli/)を使用した場合：

```bash
$ aws s3 ls
An error occurred (InvalidAccessKeyId.NotFound) when calling the ListBuckets operation: The AWS Access Key Id you provided does not exist in our records.
```

AWS [SDK](/glossary/sdk/)（Python）での[JSON](/glossary/json/)[レスポンス](/glossary/レスポンス/)例：

```json
{
  "Error": {
    "Code": "UnrecognizedClientException",
    "Message": "The security token included in the request is invalid"
  }
}
```

[API](/glossary/api/) Gateway経由での呼び出しでの[エラー](/glossary/エラー/)：

```bash
HTTP/1.1 401 Unauthorized
{
  "message": "Unauthorized"
}
```

## よくある原因と解決手順

### 原因1：アクセスキーが無効または存在しない

AWS [IAM](/glossary/iam/)ユーザーのアクセスキーが削除されたり、誤入力されたりしている場合に発生します。特に手動で[環境変数](/glossary/環境変数/)やConfigファイルに設定した場合は打ち間違いが多いです。

**Before（[エラー](/glossary/エラー/)が起きる[コマンド](/glossary/コマンド/)）：**

```bash
export AWS_ACCESS_KEY_ID="<your-access-key-id>"
export AWS_SECRET_ACCESS_KEY="<your-secret-access-key>"
aws ec2 describe-instances
```

**After（修正後）：**

```bash
# IAMコンソールで現在のアクセスキーを確認
# または新しいアクセスキーを生成してから設定
export AWS_ACCESS_KEY_ID="<your-access-key-id>"
export AWS_SECRET_ACCESS_KEY="<your-secret-access-key>"
aws ec2 describe-instances
```

確認手順は以下の通りです。AWSマネジメントコンソールで[IAM](/glossary/iam/)→ユーザー→[セキュリティ](/glossary/セキュリティ/)認証情報タブを開き、アクセスキーが有効（Active）な状態であることを確認してください。無効な場合は新しいアクセスキーを生成する必要があります。

### 原因2：セッショントークンの有効期限切れ

AssumeRoleや一時的な認証情報を使用している場合、セッショントークンには有効期限があります（デフォルト1時間）。時間が経過すると401[エラー](/glossary/エラー/)が発生します。

**Before（[トークン](/glossary/トークン/)期限切れの状態）：**

```python
import boto3
from datetime import datetime

# 1時間前に取得したセッション認証情報を使用
session_token = "AQoDYXdzEJr...(expired)"
client = boto3.client(
    's3',
    aws_access_key_id='ASIAJ...',
    aws_secret_access_key='...',
    aws_session_token=session_token
)
response = client.list_buckets()  # 401 エラー
```

**After（[トークン](/glossary/トークン/)を再取得）：**

```python
import boto3

# IAMロールを持つEC2インスタンスまたはECSタスクでは自動更新される
# または AssumeRole を再実行してトークンを更新
sts_client = boto3.client('sts')
assumed_role = sts_client.assume_role(
    RoleArn='arn:aws:iam::<account-id>:role/<role-name>',
    RoleSessionName='my-session'
)

credentials = assumed_role['Credentials']
client = boto3.client(
    's3',
    aws_access_key_id=credentials['AccessKeyId'],
    aws_secret_access_key=credentials['SecretAccessKey'],
    aws_session_token=credentials['SessionToken']
)
response = client.list_buckets()  # 成功
```

セッション認証情報の有効期限は`credentials['Expiration']`で確認できます。定期的にAssumeRoleを再実行して新しい[トークン](/glossary/トークン/)を取得するか、[IAM](/glossary/iam/)[ロール](/glossary/ロール/)付きのEC2・ECSリソースで実行する場合は自動更新に任せてください。

### 原因3：IAMポリシーが適切に設定されていない

認証情報自体は有効でも、実行しようとしている操作に対する[IAM](/glossary/iam/)[ポリシー](/glossary/ポリシー/)が不足している場合があります。この場合でも401[エラー](/glossary/エラー/)が返される（より正確には403Forbiddenとの区別が曖昧）ことがあります。

**Before（権限不足の[ポリシー](/glossary/ポリシー/)）：**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/public/*"
    }
  ]
}
```

上記の[ポリシー](/glossary/ポリシー/)で`s3:ListBucket`を実行しようとすると401に近い扱いになります。

**After（必要な操作を許可）：**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
```

AWSポリシーシミュレーター（[IAM](/glossary/iam/)[コンソール](/glossary/コンソール/)→ユーザー/[ロール](/glossary/ロール/)選択→ポリシーシミュレーター）を使用して、実際の操作が許可されているか事前に確認できます。

### 原因4：AWS CLI設定ファイルの破損または不完全な設定

`~/.aws/credentials`や`~/.aws/config`ファイルが不正な形式か、必須フィールドが不足している場合があります。

**Before（不完全なcredentialsファイル）：**

```ini
[default]
aws_access_key_id = <your-access-key-id>
# aws_secret_access_key が記載されていない
```

**After（正しい形式）：**

```ini
[default]
aws_access_key_id = <your-access-key-id>
aws_secret_access_key = <your-secret-access-key>
region = ap-northeast-1
```

[設定ファイル](/glossary/設定ファイル/)の形式を確認するには、以下の[コマンド](/glossary/コマンド/)で現在の設定を表示できます：

```bash
aws configure list
aws sts get-caller-identity  # 認証情報が有効かテスト
```

## AWSツール固有の注意点

**[API](/glossary/api/) Gatewayの401[エラー](/glossary/エラー/)：**
[API](/glossary/api/) Gatewayに [IAM](/glossary/iam/)[認証](/glossary/認証/)を設定している場合、[リクエスト](/glossary/リクエスト/)に`Authorization`[ヘッダー](/glossary/ヘッダー/)が正しく含まれていない、または署名が無効な可能性があります。AWSCLIや[SDK](/glossary/sdk/)を使用すると自動で署名されますが、curlや外部ツール使用時はAWSSignatureVersion4で手動署名が必要です。

**Cognitoユーザープール[認証](/glossary/認証/)の場合：**
OAuth2.0またはOpenIDConnectの[ID](/glossary/id/)[トークン](/glossary/トークン/)検証で401が発生する場合、[トークン](/glossary/トークン/)の有効期限切れやクライアント[ID](/glossary/id/)の不一致が原因です。`aws cognito-idp initiate-auth`で[トークン](/glossary/トークン/)を再取得するか、リフレッシュトークンで更新してください。

**Lambda実行時の認証情報：**
EC2やECS内のLambda関数からAWSリソースにアクセスする場合、[IAM](/glossary/iam/)実行[ロール](/glossary/ロール/)（Execution Role）に必要な[ポリシー](/glossary/ポリシー/)が付与されていることを確認してください。[ロール](/glossary/ロール/)はLambda関数作成時に指定される必要があります。

**CloudFormationやスタック操作：**
[IAM](/glossary/iam/)ユーザーがCloudFormationスタックを作成・更新する場合、そのユーザーに`cloudformation:CreateStack`、`iam:PassRole`などの[ポリシー](/glossary/ポリシー/)が必要です。不足していると401ではなく403Forbiddenになることもあります。

## それでも解決しない場合

**AWS [CLI](/glossary/cli/)デバッグログの確認：**

```bash
aws s3 ls --debug 2>&1 | grep -i "authorization\|credential"
```

この[コマンド](/glossary/コマンド/)で認証情報の送信状況を詳しく確認できます。

**CloudTrailで[リクエスト](/glossary/リクエスト/)失敗の詳細を確認：**
AWSマネジメントコンソールのCloudTrailサービスで、失敗した[API](/glossary/api/)コール（errorCode：「InvalidAccessKeyId」など）の詳細[ログ](/glossary/ログ/)を確認してください。リージョンやソース[IPアドレス](/glossary/ipアドレス/)などの情報が記録されています。

**STS[エンドポイント](/glossary/エンドポイント/)の動作確認：**

```bash
aws sts get-caller-identity --profile <your-profile>
```

この[コマンド](/glossary/コマンド/)で成功すれば、基本的な認証情報は有効です。失敗する場合は認証情報そのものが無効です。
