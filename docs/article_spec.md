# 記事仕様（article_spec）

errorlog.jp のエラー解説記事の品質指針。

このファイルは品質の基準を示す指針であり、生成挙動を直接制御するものではない。
実際の生成は各スクリプトのシステムプロンプトが、機械チェックは lint_articles.py が担う。
記事を書く・見直すときは、まずこの指針に立ち返る。

このサイトの核心的価値は技術的正確性である。構成の見栄えより、書かれている内容が
正しいことを最優先する。以下、第1部（正確性の絶対原則）が最も重要で、第2部以降の
構成・形式はすべてこれに従属する。

---

## 第1部：正確性の絶対原則（最優先・例外なし）

記事の技術的内容は、次の原則をすべて満たさなければならない。形式が整っていても、
この原則に反する記事は公開しない。

### 1. 推測で書かない

原因・対処・コード・設定・数値は、確認できる事実だけを書く。確認できないことを
「おそらくこうだろう」と、もっともらしく書いてはならない。

具体的な値（レート制限の回数、タイムアウト秒数、設定キー名、API のフィールド名、
ヘッダー名、コマンドの構文）は、必ず公式ドキュメントで確認したものだけを書く。
記憶やそれらしさで補わない。確認できないなら、その値は書かず、
「公式ドキュメントで確認すること」と促すにとどめる。

### 2. ソースを確認する

参照する URL は、実際にアクセスして実在と内容を確認したものだけを使う。
スラッグ（URL 末尾の文字列）がそれらしくても、中身が無関係なページを引用しない。
取得できない・内容を確認できないソースは引用しない。

引用するソースが古い場合（数年前のバージョンを前提にした記事など）は、
その旨を明記する。古くても現在に通用する内容なら、なぜ通用するかを一言添える。
通用しないなら引用しない。

### 3. エラーコードの因果を正しく扱う

500 番台（500, 502, 503, 504）はサーバー側のエラー、4xx 番台（400, 401, 403,
404, 422, 429）はクライアント側のエラーである。この因果を混同しない。

たとえば、無効な認証は 401、権限不足は 403、不正なリクエスト形式は 400 や 422 で
あって、これらを 500 や 502 の主要原因として挙げてはならない。クライアント側の
問題が 500 番台を誘発するのは例外的なケースであり、そう書く場合は例外だと明示する。

### 4. 仕様変化に注意する

ライブラリやサービスの仕様は変わる。バージョン番号を本文に固定で書くと、すぐ古く
なる。特定バージョンに依存する記述は、できるだけ「現行の標準的な書き方」という
様式で書き、バージョン番号の固定記載を避ける。SDK のコード例は、執筆時点の現行
API に従う。

## 第2部：記事の骨格（推奨。エラーの性質に応じて選ぶ）

構成は画一的に固定しない。エラーの性質に最も適した形を選ぶ。ただし、質の高い
記事には共通して次の要素がある。これらを骨格として、過不足なく構成する。

### 必ず含めるもの

- **冒頭まとめ（BLUF）**：記事の最初に、結論を先出しする。そのエラーの主な原因と、
  調査の出発点を、数文で要約する。読者が最初に全体像をつかめるようにする。
- **エラーの概要**：そのエラーコードが何を意味するか。近いエラーコード（401 と 403、
  403 と 404 など）との違いに触れ、読者が自分の状況を正しく分類できるようにする。
- **原因と解決策**：原因を体系的に挙げ、それぞれに対処を示す。原因の数は、エラーの
  実態に合わせる（無理に数を揃えない）。
- **確認・切り分けの手順**：読者が自分の状況を診断できる、具体的なコマンドや手順。
  「それでも解決しない場合」として、確認コマンドをまとめてもよい。
- **免責事項**：本文末尾に、定型の免責事項フッターを置く（第4部参照）。

### エラーの性質に応じて選ぶもの

- **診断の枠組み**：原因の切り分けにログや状態確認が重要なエラー（権限・設定・
  接続系）では、「まず何を確認するか」を原因解説の前に置くと有用。ログの文言と
  原因の対応を示すと、読者が切り分けやすい。
- **解決策の早見表**：原因が多い場合、表で一覧できると便利。
- **ツール固有の注意点**：そのツール特有の落とし穴（ワーカーユーザーの違い、
  ディストリ差、版による挙動差など）をまとめると深みが出る。
- **Editor's Note**：実際の報告事例を、実在・内容を確認したうえで引用する。
  読者を実例に導く独自性のある要素（※ 現在、引用の信頼性を担保する仕組みが
  整うまで生成停止中）。

### コード・対処の示し方

エラーの性質に応じて、適した形式を選ぶ。

- コードの書き方そのものが原因のエラー（API の使い方ミス、不正な構文など）では、
  **Before（問題のあるコード）と After（修正後）の対比**が有効。
- 設定や権限の操作が対処になるエラー（権限不足、SELinux、設定ミスなど）では、
  **診断コマンド → 対処コマンド**の流れが自然。Before/After を無理に当てはめない。

どちらの場合も、対処の後に、解決したことを確認する手順（確認コマンドや期待される
出力）を添えると、読者が結果を検証できる。

## 第3部：frontmatter と SEO に必要な情報

frontmatter には、サイトの動作（taxonomy、検索表示）に必要なフィールドを、
正しく設定する。

- `title`：記事タイトル（必須）。
- `errorCode`：エラーコード（必須）。taxonomy のキーになる。
- `service`：ツール名（必須）。
- `tags`：タグ（必須）。
- `description`：検索結果に表示される説明文。そのエラーと記事の要点を簡潔に。
  重複表現（「。。」など）を含めない。
- `date` / `lastmod`：公開日・更新日。記事を実際に修正したときは lastmod を更新する。
- `conclusion`：冒頭まとめに対応する要約（推奨）。
- `top_queries`：想定する検索クエリ（任意）。

## 第4部：表記と体裁

- 文体はですます調。断定を避けすぎず、確認できる事実は明確に書く。
- H1 タイトルは本文に含めない（frontmatter の title が使われる）。
- コードブロックには言語名を付ける。
- プレースホルダーは `<your-xxx>` 形式を使う。
- 認証トークン・API キー・パスワードなどの秘密情報は、実値を書かない。
  必ずプレースホルダーにする。
- 「まとめ」という名の締めセクションは置かない（冒頭まとめが結論を担う）。
- 逃げ腰な表現（「直接的なエラーを扱うわけではない」など、エラーに正面から
  答えない言い回し）を使わない。
- 本文末尾に、次の免責事項フッターを置く。

```
---

*免責事項：本記事の内容は、執筆時点の公開情報をもとに作成したものです。ソフトウェアの仕様は予告なく変更されることがあります。最新の情報は各ツールの公式サポートページをご確認ください。本記事の情報を利用した結果生じたいかなる損害についても、著者および運営者は責任を負いかねます。*
```

## 補足：この指針と機械チェックの関係

- 記事の**構造**（セクションの順序や数）は、この指針が示すとおり、エラーの性質に
  応じて柔軟に選ぶ。機械的な構造チェックでは縛らない。
- 機械チェック（lint）が担うのは、frontmatter の必須フィールド、免責事項の有無、
  秘密情報の不在、不適格表現の不在、Editor's Note の引用元の安全性など、
  データの整合と安全に関わる部分に限る。
- 技術的内容の**正確性**（第1部）は、機械チェックでは判定できない。公式ドキュメント
  との照合という、人の確認（または確認を経た工程）で担保する。これが品質の要である。
